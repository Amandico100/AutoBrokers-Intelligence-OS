# 39A4.0 — WhatsApp Architecture Recon

> **Status:** recon read-only concluído · **somente docs criados** (relatório + SQL SELECT-only) · nenhum código/SQL/schema alterado, nenhuma mensagem enviada, nenhum segredo lido, nenhuma API externa.
> **Data:** 2026-06-08 · **Modelo:** Claude Opus 4.8 · **Branch:** main · **Tipo:** auditoria + decisão estratégica.

## 1. Resumo executivo
O runtime novo (Smith) já tem um **caminho de WhatsApp funcional via Z-API**, multi-tenant por empresa/agente, integrado a usuários/conversas/agentes/billing/handoff, com **DRY_RUN**. A estrutura **legada (ResultVision)** — auditada read-only localmente — implementa WhatsApp de forma **rica em domínio mas pesada**: **Evolution API orquestrada por n8n** (workflows WF01/WF02 com policy-guards) + um **motor de assistência de seguradora** (`insurer-whatsapp`), em um stack diferente (Express/Drizzle/Neon/Replit + Twilio). Recomendação: **Caminho C** — manter o **Vault como camada de produto** + criar um **provider adapter**, usando **Z-API/Smith como primeiro provider**; tratar o motor `insurer-whatsapp` do ResultVision como **referência de domínio** (não base de runtime) e o n8n como **adapter opcional** (ADR-001 §14), nunca orquestrador. Gaps a corrigir: **`integrations.token`/`client_token` em plaintext (P0)** e **autenticação do webhook inbound**.

## 3. Arquitetura WhatsApp no Smith Runtime atual
Z-API ponta a ponta, no backend FastAPI: **inbound** via webhook (background task) → resolve integração por `connectedPhone` → usuário/lead → conversa/sessão → gate de saldo → IA (LangGraph) → salva → **outbound** via Z-API REST. Provider-agnóstico na intenção (`VALID_PROVIDERS`), Z-API na execução. Sem QR no app (a instância é pareada no painel Z-API).

## 4. Tabelas envolvidas
- **`integrations`**: `company_id`, `provider` (`z-api`), `identifier` (= connectedPhone), `token`, `instance_id`, `base_url`, `client_token`, `is_active`, `agent_id`, `buffer_*`. Credencial por empresa/agente.
- **`conversations`** (`status`, `channel`='web'/'whatsapp', `agent_id`, `human_handoff_reason`, `session_id`), **`messages`**, **`conversation_logs`**, **`users_v2`** (lead por telefone), **`agents`**, **`company_credits`/`token_usage_logs`** (gate/cobrança). Vault: `tenant_connections`/`connector_templates`/`approval_requests`/`vault_audit_log` (39A1).

## 5. Services envolvidos
- `services/integration_service.py` — `get_integration_by_phone` (inbound), `get_whatsapp_integration(company_id, agent_id)` (outbound, estrito), `get_or_create_user`.
- `services/whatsapp_service.py` — `send_message/send_audio/send_image` (Z-API REST, DRY_RUN).
- `services/message_buffer_service.py` — buffer/debounce de mensagens (Redis).
- `services/langchain_service.py` — IA. `services/audio_service.py` — áudio. `services/billing_service.py` — `has_sufficient_balance`.

## 6. Endpoints envolvidos
- **Backend:** `POST /webhook` (Z-API inbound; processamento em background) + rota admin (set webhook) protegida por `require_master_admin`. Outbound é interno (service), não endpoint público.
- **Next:** `app/api/admin/integrations/route.ts` (admin CRUD), `app/api/n8n/route.ts` (passthrough legado), Vault `app/api/vault/*` (39A2).

## 7. Fluxo inbound atual
`process_whatsapp_message_background(payload)`: `connectedPhone` → `get_integration_by_phone` → `company_id`/`agent_id` → `get_or_create_user(phone, company_id)` → `session_id=whatsapp:{phone}:{company}:{agent}` → se `HUMAN_REQUESTED` para; senão **gate de saldo** → `LangChainService.process_message` → salva mensagens → `whatsapp_service.send_message`. Telefone sempre logado mascarado (`...1234`).

## 8. Fluxo outbound atual
`whatsapp_service.send_message(to, text, integration)` → `POST {base_url}/{instance_id}/token/{token}/send-text` (+ `Client-Token` header se houver). Integração obtida por `get_whatsapp_integration(company_id, agent_id)` — **estrito, sem fallback** (aborta se não casar exatamente com o agente, para não cruzar conversas). **DRY_RUN** simula (retorna True sem HTTP).

## 9. Como empresa/agente são identificados
**Inbound:** por `connectedPhone` = `integrations.identifier` → integração → `company_id`+`agent_id`. **Outbound:** por `company_id`+`agent_id` com match exato (sem fallback). Isolamento forte; risco é só se duas integrações ativas tiverem o mesmo `identifier` (улnão deve ocorrer).

## 10. Como usuários/leads são criados
`get_or_create_user(phone, company_id, name)`: procura em `users_v2` por (phone, company) → por email `{{phone}}_{{company}}@whatsapp.smith.ai` → senão cria **lead** (status `lead`, CPF hash determinístico, nome do WhatsApp ou "Usuário WhatsApp"). Email/CPF únicos por telefone+empresa (evita colisão entre empresas).

## 11. Como conversas são criadas/atualizadas
Por `session_id` (`whatsapp:{phone}:{company}:{agent}`) — busca/cria conversa, salva mensagens (`messages`), atualiza preview/unread; `status` controla handoff humano. Reuso do mesmo modelo do chat web (channel diferencia).

## 12. Como DRY_RUN funciona
Env **global** `settings.DRY_RUN` (`core/config.py`, default False). Quando true: `get_integration_by_phone` acha a integração **real** (company/agent verdadeiros) mas sobrescreve `token`/`instance_id` por `dry-run-*`; `send_message` retorna True sem chamar a Z-API. Ótimo para ambiente de teste seguro; **não** é controle por requisição/conexão (é tudo-ou-nada do ambiente).

## 13. Como Z-API está modelado
Credencial por empresa/agente em `integrations` (token/instance/base_url/client_token). Sem QR no app (instância pareada no painel Z-API). Envio por REST direto. `client_token` opcional vai no header. `base_url` default `https://api.z-api.io/instances`. Provider string `z-api`.

## 14. Como Evolution/n8n aparece na estrutura antiga
No **ResultVision** (legado, stack Express/Drizzle/Neon/Replit + Twilio + Gemini):
- **n8n como orquestrador:** `server/lib/n8n.ts`, `scripts/n8n/*`, `test-n8n.ts`, `server/lib/n8n/wf01-policy-guard.ts`+`wf02-policy-guard.ts`.
- **WF01 = "Atendimento WhatsApp (Even) v2"** (`INSTRUCOES/WF01…json`) — "Even" = **Evolution**. WF02 = segundo workflow.
- **Evolution:** `docs/reports/2026-04-03-coverage-matrix-evolution.md` (matriz de cobertura Evolution).
- **WhatsApp inbox:** `server/lib/whatsapp-inbox/*` (outbound-dispatch, canonical inbox), `whatsapp-inbox-media.ts`, client `settings-whatsapp.tsx`.
- **Motor de assistência seguradora:** `server/lib/insurer-whatsapp/*` (orchestrator/parser/recipes/runtime/schema/csat) + plano `docs/plans/2026-04-09-insurer-whatsapp-assistance-engine.md` + `sql/…phase7b.sql`. Ponte de voz: `voice-whatsapp-bridge.ts` (Twilio).
> Não abri o JSON do WF01 nem `.env`/credenciais. Mapeamento por nomes de arquivo + `package.json`.

## 15. O que da estrutura antiga parece aproveitável
**Como referência de DOMÍNIO (não runtime):** o motor `insurer-whatsapp` (recipes/orchestrator/parser de assistência por seguradora), a lógica de **policy-guards** (WF01/WF02 = Action Safety), o **canonical inbox** e a matriz de cobertura Evolution. Esses conceitos informam o módulo **Atendimentos** e o futuro provider Evolution — extraídos/curados, **não copiados** (ADR-001/ADR-002 §13/§14).

## 16. O que da estrutura antiga deve ser descartado
Como **base de runtime**: o stack Express/Drizzle/Neon/Replit, o acoplamento a **n8n como orquestrador**, worktrees/duplicação, e qualquer credencial embutida em workflow. Não migrar bruto; não adotar n8n como núcleo (contra ADR-001 §14).

## 17. Estado da criptografia de tokens
**`integrations.token` e `client_token` estão em PLAINTEXT** no runtime novo: `whatsapp_service` lê `integration.get("token")` cru e monta a URL Z-API; `integration_service` nunca descriptografa. O `EncryptionService` (Fernet/AES) existe e é usado para MCP/LLM keys, **mas não para `integrations`**. → **P0:** migrar para cifrado (reuso de `EncryptionService`) antes de expandir conectores reais (ADR-002 §18).

## 18. Relação com o Vault novo
O Vault (39A1/39A2/39A3) é a camada de produto. Uma `tenant_connections` (template `whatsapp_zapi`) deve **referenciar** a `integrations` existente: `technical_ref_type='integration'`, `technical_ref_id=integrations.id`, `encrypted_secret_ref` apontando para o segredo cifrado — **sem duplicar o token**. Permissões (`permission_grants`) e aprovações (`approval_requests`) governam o uso por Auxiliares/Atendimentos.

## 19. Relação com HITL/aprovações
Outbound real deve passar por `approval_requests` quando a permissão exigir (`requires_approval`). Hoje o webhook responde inbound automaticamente (atendimento); para **ações iniciadas por Auxiliar** (ex.: cobrança), o envio só deve ocorrer **após aprovação** — o adapter de envio deve checar o HITL antes de chamar `whatsapp_service.send_message`.

## 20. Relação com Atendimentos
Inbound WhatsApp já alimenta `conversations` (canal whatsapp) + handoff humano — base do módulo Atendimentos. O motor `insurer-whatsapp` legado é a referência para assistência/sinistro/corredores. Portais de seguradora seguem alto risco (ADR-002 §25), fora deste escopo.

## 21. Relação com Auxiliares
Auxiliares (ex.: Cobrança/Follow-up) usarão o conector WhatsApp via Vault: **rascunho** (sem envio) → `approval_requests` → aprovação → envio pelo provider adapter (Z-API). O Auxiliar não recebe credencial; usa permissão de uso.

## 22. Riscos técnicos
- **Plaintext token (P0).** **Webhook inbound sem assinatura** (confiar só em `connectedPhone`/URL) → possível injeção de payload (P1). **DRY_RUN global** (não por conexão) → cuidado em produção. **Provider único** implementado (Z-API) apesar do lookup agnóstico → adapter formaliza isso. Legado n8n = acoplamento/tech-debt se adotado.

## 23. Riscos LGPD/segurança
PII (telefones, conteúdo de conversa) — já há mascaramento de telefone em logs (bom). Segredos não podem vazar ao client (ADR-002 §18) → cifrar `integrations`. Isolamento multi-tenant por `company_id`/`agent_id` (forte no outbound). Não ingerir intake bruto/WF01 com credenciais. Aprovação humana antes de envio externo.

## Comparação real: Smith/Z-API vs Evolution/n8n legado
| Eixo | Smith / Z-API (novo) | Evolution / n8n (ResultVision legado) |
|---|---|---|
| **Maturidade técnica** | Integrado ao runtime oficial (FastAPI/LangGraph/Supabase), em uso | Maduro em domínio (assistência), mas Replit/Express/Drizzle, fora do runtime |
| **Multi-tenant** | Nativo: `integrations` por company/agent | Via workflows n8n; multi-tenant mais frágil/global |
| **Isolamento empresa/agente** | Forte: lookup estrito **sem fallback** | Depende da config do workflow (risco de cruzar) |
| **Risco de sujeira técnica** | Baixo (um runtime, limpo) | Alto (worktrees/duplicação, plugins Replit, stack distinto) |
| **QR Code / conexão** | Sem QR (Z-API pareado no painel) | Evolution normalmente faz QR; provável no legado (UI `settings-whatsapp`), não confirmado sem abrir |
| **Inbound** | Webhook FastAPI nativo → IA | n8n WF01 → canonical inbox |
| **Outbound** | `whatsapp_service` Z-API (estrito, DRY_RUN) | n8n outbound-dispatch (canonical) via Evolution |
| **Logs** | `conversation_logs`/`token_usage_logs`/`system_logs` + sanitização | Próprios (Drizzle), fora do runtime |
| **Segurança** | Boa estrutura; falta cifrar `integrations` | Creds no n8n/workflow (risco de embutidas) |
| **Criptografia** | `EncryptionService` existe (aplicar a integrations) | Não avaliado (não abrir credenciais) |
| **Dependência de n8n** | Nenhuma (n8n só passthrough legado) | **Alta** (orquestrador) — contra ADR-001 §14 |
| **Facilidade de manutenção** | Maior (um runtime coeso) | Menor (Express+n8n+Evolution+Twilio) |
| **Escala p/ Auxiliares/Atendimentos** | Já integra agents/billing/HITL/Vault → melhor | Engine de assistência valioso como **referência**, não base |

**Veredito:** Smith/Z-API vence como **base de runtime**; o legado vence como **fonte de domínio** (insurer-whatsapp, policy-guards). Por isso, **C**.

## 24. Recomendação A/B/C
- **A) Z-API/Smith como base oficial.** Prós: já pronto, limpo, multi-tenant, DRY_RUN. Contras: sem a abstração de provider + governança do Vault para multi-provider/HITL.
- **B) Evolution/n8n legado como base oficial.** Prós: domínio rico, possível QR. Contras: tech-debt, n8n-orquestrador (contra ADR-001 §14), stack distinto, multi-tenant/governança mais fracos.
- **C) Vault + provider adapter, Z-API primeiro.** Prós: escalável, Z-API agora + Evolution depois pelo mesmo adapter, Vault como produto/governança, HITL, reaproveita domínio legado como referência. Contras: exige uma camada fina de abstração.

## 25. Caminho recomendado: **C**
Vault (produto) + **provider adapter** (`WhatsAppProvider` com `ZApiProvider` primeiro, `EvolutionProvider` depois) sobre o que já existe (`integrations` + `whatsapp_service`), com **segredo cifrado** e **HITL** antes de envio iniciado por Auxiliar. Confirma a hipótese do fundador e ADR-001 §15.
