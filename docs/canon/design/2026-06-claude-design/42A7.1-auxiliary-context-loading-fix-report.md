# 42A7.1 — Auxiliary Context Loading Fix Report

> **Status:** concluído · py_compile verde · `git diff --check` limpo · **sem RAG/Qdrant/MinIO/upload, sem WhatsApp/executores, sem schema/SQL/migration, sem prompt de produção, sem web search**.
> **Data:** 2026-06-12 · **Modelo:** Claude Opus 4.8 · **Branch:** main

## 1. Causa provável (confirmada por leitura de código)
`load_tenant_auxiliaries_for_context` selecionava **`display_name`**:
`.select("id, slug, status, name, display_name, config")`.
A tabela `tenant_auxiliaries` **não tem** a coluna `display_name` (confirmado pelo founder no Supabase; o factory `TENANT_FALLBACK_COLS` e o install nunca a garantem). No PostgREST/Supabase, selecionar uma coluna inexistente faz a **query inteira falhar** (HTTP 400). O `try/except` engolia o erro e retornava `[]` → o bloco `[AVAILABLE AUXILIARIES]` **nunca era injetado** → o Core respondia por inferência/prompt e podia **inventar** auxiliares. **Causa confirmada.**

## 2. Correção feita
Arquivo: `backend/app/agents/auxiliary_context.py`.
1. **Query corrigida:** seleciona `id, template_id, slug, status, name, config` (sem `display_name`). Mais: **fallback resiliente a schema** — se o select explícito falhar por qualquer motivo, refaz com `select("*")` e filtra em memória (nunca mais quebra por nome de coluna).
2. **Templates relacionados:** `_attach_templates` busca em `auxiliary_templates` (por `template_id`, via `IN`) apenas colunas seguras (`id, slug, name, short_description, description, default_config, requires_human_approval, uses_external_actions, execution_mode, trigger_type`) e anexa em `row['_template']`. (Usado só para extrair/normalizar o contrato; **não** renderiza `default_config` inteiro.)
3. **Normalização melhorada** (`normalize_auxiliary_contract_for_context`): contrato vem de **tenant `config.contract` → template `default_config.contract` → inferência mínima** (usando flags do template `requires_human_approval`/`uses_external_actions` + runtime, espelhando `lib/auxiliaries/contract.ts`). Nome = `row.name` → título conhecido → `template.name` → slug (sem depender de `display_name`). `goal` cai para `template.short_description`/`description` quando o contrato não traz.
4. **Instruções mais rígidas** no bloco: listar **apenas** os auxiliares do bloco; **não inventar** auxiliares; se não estiver listado, dizer que não consta como instalado; ideias futuras só rotuladas como sugestões, não instaladas.

## 3. Como preserva o Smith
Mudança **isolada** no helper de awareness. **Nada** em SearchService/Qdrant/MinIO/upload/ingestão/RAG, WhatsApp/executores/approval/billing, schema/SQL/migration, prompt no banco ou web search. `include_global` inalterado. Agentes não-Core ou sem trigger seguem idênticos. Erro em qualquer ponto degrada para `[]`/`""` sem quebrar o chat.

## 4. Como evita bloat
Mantido do 42A7: só Core + trigger; ≤8 itens; linha curta por item; `goal`/`when_to_use` truncados (≤90); **nunca** `config`/`default_config`/`system_prompt`/`permissions` inteiros. O template é lido só para extrair campos curtos do contrato.

## 5. Como evita inventar Auxiliares
Agora o bloco **carrega de fato** os instalados (query corrigida) e as instruções proíbem inventar: "list ONLY the auxiliaries in this block", "Do not invent installed auxiliaries", "If an auxiliary is not listed here, say it is not currently shown as installed". Ideias futuras só como sugestões rotuladas.

## 6. Sanitização
Só campos whitelisted do contrato são lidos. Validado por teste: chave `secret` dentro do `default_config.contract` do template **não** aparece no bloco. Sem token/api_key/credential/PII/payload.
