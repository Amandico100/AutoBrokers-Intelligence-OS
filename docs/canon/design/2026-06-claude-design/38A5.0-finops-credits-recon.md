# 38A5.0 — Auxiliaries FinOps / Credits Recon

> **Status:** recon read-only concluído · **somente docs criados** (relatório + SQL SELECT-only) · nenhum código/SQL/schema alterado, nenhum acesso ao banco.
> **Data:** 2026-06-08 · **Modelo:** Claude Opus 4.8 · **Branch:** main · **Tipo:** auditoria + proposta (sem implementação).

## 1. Resumo executivo
A cobrança do AutoBrokers é **assíncrona e desacoplada**: toda chamada de IA grava `token_usage_logs` (`billed=false`) e um **worker Celery** (`process_unbilled_usage`) cobra **todos** os logs não cobrados, debitando `company_credits` e registrando `credit_transactions`. O Auxiliar de Resumo (38A2) **já grava `token_usage_logs` com `service_type='auxiliary_run'` em caso de sucesso** → portanto **já é cobrado** pelo pipeline existente, sem nada novo. Falhas não são cobradas. O **único gap real** é a **ausência de verificação de saldo antes de executar** o Auxiliar (o chat já faz isso via `has_sufficient_balance`). **Recomendação: Caminho A — reutilizar 100% o billing existente**, adicionando depois apenas o gate de saldo (também reuso). Sem tabela nova, sem SQL de schema.

- **Backend:** `services/usage_service.py`, `services/billing_service.py`, `workers/billing_core.py`, `workers/billing_tasks.py`, `core/callbacks/cost_callback.py`, `api/billing.py`, `api/auxiliaries.py`, `api/webhook.py`, `api/documents.py`.

- **Não aberto:** pastas externas (ResultVision/INTAKE/AGENT_OS/QUARENTENA); sem conexão ao Supabase.

## 3. Tabelas encontradas (billing/créditos/uso)
- **`company_credits`** — `company_id`, **`balance_brl numeric(10,4)`**, `alert_80_sent`, `alert_100_sent`, `updated_at`. Saldo atual da empresa.
- **`credit_transactions`** — `company_id`, `agent_id`, `type` (`subscription|topup|consumption|refund|bonus`), `amount_brl` (negativo em consumo), `balance_after`, `model_name`, `tokens_input/output`, `description`, `stripe_payment_id`, `created_at`.
- **`token_usage_logs`** — `company_id`, `agent_id`, **`service_type`**, `model_name`, `input_tokens`, `output_tokens`, `total_cost_usd`, `details jsonb`, **`billed`**, `billed_at`, cache tokens. Fonte bruta de uso/custo (USD).
- **`subscriptions`** — plano/status/período/Stripe. **`plans`** — `price_brl`, `display_credits`, `sell`/preço. **`llm_pricing`** — preços por modelo (input/output, `sell_multiplier`). **`payment_history`** — pagamentos.

## 4. APIs/backend services encontrados
- **`usage_service`**: `track_cost_sync(service_type, model, input_tokens, output_tokens, company_id, agent_id, details)` → insere `token_usage_logs` (USD, `billed=false`); `calculate_cost(model, in, out)`; `calculate_and_debit_client(...)` (débito inline — **não usado pelo chat**, ver §5).
- **`billing_core`/`billing_service`**: `get_company_balance`, `add_credits`, `reset_credits`, **`debit_credits`** (RPC atômica `debit_company_balance` + `credit_transactions` + alertas 80/100%), **`has_sufficient_balance(company_id)`** (cache 30s), Stripe.
- **`workers/billing_tasks.py`**: `process_unbilled_usage` (periódico) e `process_company_billing` (on-demand) — leem `token_usage_logs` `billed=false`, agrupam por `(company_id, agent_id, model_name)`, convertem USD→BRL e debitam; marcam `billed=true`.
- **`api/billing.py`**: `GET` saldo/uso do tenant (para UI).

## 5. Como o chat atual registra uso/custo
O chat **não debita inline**. Um `CostCallback` (`core/callbacks/cost_callback.py`) chama `usage_service.track_cost_sync(...)` → grava `token_usage_logs(billed=false)`. O **worker Celery** cobra depois em lote (`debit_credits`), marcando `billed=true`. Antes de invocar IA, o sistema verifica `has_sufficient_balance` (ex.: `webhook.py`, `documents.py`). Ou seja: **logar uso = cobrar (assíncrono); gate de saldo = pré-checagem**.

## 6. Como auxiliary_runs registra uso/custo hoje
`backend/app/api/auxiliaries.py` (38A2/38A2.1), **em caso de sucesso**, chama `track_cost_sync(service_type='auxiliary_run', company_id, model='gpt-4o-mini', input_tokens, output_tokens, details={auxiliary, run_id, tenant_auxiliary_id})` → grava `token_usage_logs(billed=false)`. Também grava `cost_usd` + `token_usage` **na própria `auxiliary_runs`** (display/log). Em **falha**, `_fail_run` **não** chama `track_cost_sync` (não loga uso). Conclusão: **cada execução bem-sucedida gera um log de uso que o worker já cobra**; `auxiliary_runs.cost_usd` é apenas espelho.

## 7. Gaps entre Auxiliares e billing atual
- **G1 (principal):** **não há gate de saldo** antes de executar o Auxiliar. O chat bloqueia com `has_sufficient_balance`; o run de Auxiliar não. Empresa sem saldo pode executar e ficar negativa.
- **G2 (visibilidade):** a UI de Auxiliares não mostra saldo/créditos nem mensagem de "sem saldo".
- **G3 (atribuição):** `credit_transactions` não tem `service_type`; o custo de Auxiliar fica como `consumption` (model `gpt-4o-mini`, `agent_id` nulo). A atribuição "isto foi um Auxiliar" vive em `token_usage_logs.service_type='auxiliary_run'` (suficiente para relatórios).
- **Não-gap:** a cobrança em si já acontece (não precisa de bridge novo).

## 8. Riscos de cobrança dupla
**Baixo.** A cobrança é **idempotente pelo flag `billed`** — cada linha de `token_usage_logs` é cobrada uma única vez. Como o Auxiliar só loga **um** registro por execução **bem-sucedida**:
- **Retry/reload da UI:** cada clique que dispara uma nova execução cria uma nova `auxiliary_run` + um novo log = um **custo real distinto** (correto cobrar). Não há reprocessamento do mesmo log.
- **Reprocessamento do worker:** após `billed=true`, o log não é relido. Sem duplicação.
- Risco residual: se um futuro código chamar **tanto** `track_cost_sync` **quanto** `calculate_and_debit_client` para a mesma execução → cobraria 2x. **Não fazer isso** (manter só o caminho `track_cost_sync` → worker).

## 9. Riscos de cobrar execução com erro
**Mitigado hoje.** `track_cost_sync` só é chamado **após** o LLM responder com sucesso; falhas (`_fail_run`) não logam uso → não são cobradas. Manter essa ordem (logar custo **somente** em sucesso). Atenção futura: se a chamada LLM consumir tokens e depois falhar no parse, hoje não logamos — aceitável (não cobra), mas é uma subcontagem mínima de custo real; ok para o MVP.

## 10. Recomendação de arquitetura para débito por execução
**Manter o modelo assíncrono existente** (logar `token_usage_logs` em sucesso → worker debita). É o mesmo padrão do chat, atômico (RPC `debit_company_balance`), com alertas 80/100% e idempotência por `billed`. **Não** debitar inline no endpoint do Auxiliar (evita lógica duplicada e risco de dupla cobrança). Acrescentar somente:
- **Gate de saldo (pré-execução):** no início do run, validar `has_sufficient_balance(company_id)`; se falso, **não executar** e retornar erro amigável "sem créditos" (HTTP 402/403). Reuso direto de `billing_service` — backend, antes de criar a `auxiliary_run`.

## 11. Recomendação de MVP para cobrança/créditos
**Caminho A (reuso 100%)** — a cobrança já funciona. MVP de FinOps dos Auxiliares = **apenas**:
1. **Gate de saldo** no endpoint de run (reuso de `has_sufficient_balance`).
2. **UI:** mostrar saldo/aviso no fluxo do Auxiliar (reuso de `GET /api/billing`) e mensagem amigável quando bloqueado.
Sem novas tabelas, sem SQL de schema, sem alterar o worker. Débito de crédito específico do batch 38A4 (que cogitamos) **não é necessário** — já ocorre.

## 12. O que precisa de SQL
**Nada de DDL.** Apenas o **SQL diagnóstico SELECT-only** (`docs/sql/38A5.0-finops-diagnostics.sql`) para o fundador confirmar no Supabase: existência das tabelas/colunas/RLS, a RPC `debit_company_balance`, e principalmente o item 7/10 (logs `service_type='auxiliary_run'` e seu `billed`), provando que Auxiliares já entram na cobrança. Nenhuma migração é exigida para o MVP.

## 14. O que precisa aparecer na UI

- **Admin Global (futuro):** custo por `service_type='auxiliary_run'` (a partir de `token_usage_logs`), top empresas por uso de Auxiliares, e (futuro) atribuição por template.

## 15. O que deve ficar fora do MVP

(Tudo possível depois sobre a base atual.)

## Decisão recomendada: **A) Reutilizar 100% a estrutura existente.**
- **A (recomendado):** a cobrança por execução **já acontece** via `token_usage_logs` → worker; reuso total, idempotente, sem schema novo. **Contra:** falta só o gate de saldo (adição mínima, também reuso).
- **B (reuso parcial + bridge):** desnecessário — o "bridge" (logar uso) já existe; criaria redundância.
- **C (estrutura nova):** rejeitado — duplicaria o billing maduro do runtime, com risco de dupla cobrança e divergência de saldo.
