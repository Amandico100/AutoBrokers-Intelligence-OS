## 1. Resumo executivo
Fechamos a arquitetura: **Auxiliares = produto; Smith Agents/Subagents = runtime; Vault = governança** — agora como **lei canônica** (SPEC-002) e como **binding real** no código. Templates globais declaram runtime (`specific_executor` | `smith_agent_blueprint` | `workflow` | `none`); ao **instalar** um Auxiliar com blueprint, o sistema **cria um Agent/Subagent Smith da corretora** via o endpoint canônico `POST /api/agents/` e grava o `agent_id` em `tenant_auxiliaries.config.runtime`. O WhatsApp antigo do Agent vira **legado** (token já bloqueado em 39A4.1; caminho oficial = Vault). Nenhum schema alterado.

## 2. Estávamos criando estrutura paralela?
**Parcialmente — e foi corrigido a tempo.** O 41A criou um catálogo/governança **correto** (Auxiliares Globais), mas **sem vínculo ao runtime Smith** — havia risco real de virar um motor paralelo. Este batch **amarra** Auxiliares ao runtime Smith e proíbe motor paralelo por escrito (SPEC-002). Não há reimplementação de inteligência/execução fora do Smith.

## 3. Decisão canônica tomada
`AutoBrokers Auxiliares = produto/catálogo/UX/governança` · `Smith Agents/Subagents = runtime técnico` · `Vault = segredos/conectores/permissões/HITL/auditoria`. Auxiliar avançado **deve declarar runtime**. Template global = **blueprint** (não agent compartilhado); agent real **por empresa**.

## 7. Como o runtime blueprint é salvo
Em `auxiliary_templates.default_config.runtime` (JSON, sem schema novo):
`{ "runtime": { "kind":"smith_agent_blueprint", "agent_blueprint": { name, slug, is_subagent, allow_direct_chat, llm_provider, llm_model, agent_system_prompt } } }`. Também `specific_executor`/`workflow`/`none`. `parseRuntimeConfig` infere `specific_executor` para slugs conhecidos (`resumo-atendimentos`, `follow-up-whatsapp`).

## 8. Como o agent/subagent é criado ao instalar
`install/route.ts` lê `template.default_config.runtime`. Se `smith_agent_blueprint` (e a coluna `tenant_auxiliaries.config` existe): monta `AgentCreate` (`buildAgentCreatePayload`, slug único por sufixo) e chama **`POST /api/agents/`** com `X-Admin-API-Key` (`createAgentViaBackend`) — caminho canônico via `AgentService` (com side-effects). **Best-effort:** se falhar, a instalação não quebra (grava o erro em `config.runtime.agent_error`).

## 9. Como tenant_auxiliary vincula agent_id
Sucesso → `tenant_auxiliaries.config.runtime = { kind:'smith_agent', agent_id, created_from_template:true, template_runtime_kind:'smith_agent_blueprint' }`. **Idempotência:** se já existe instalação com `agent_id`, não cria outro agent; se existe sem `agent_id` e o template tem blueprint, vincula.

## 17. Riscos/remanescentes
- **`tenant_auxiliaries.config`:** a criação/binding do agent só ocorre se a coluna `config` existir (probe). Se ausente, instalação funciona mas sem persistir o `agent_id` (reportado). Confirmar a coluna no schema; ampliar via Architect se faltar.
- **Mapeamento `AgentCreate`:** envio conservador (company_id, name, slug, is_subagent, allow_direct_chat, llm_provider, llm_model, agent_system_prompt, is_active). Se o backend exigir/rejeitar campos, o binding falha **graciosamente** (sem quebrar a instalação) e registra `agent_error`.
- **GET legado de integrations** retorna token cifrado ao front (legado) — strip recomendado.
- Rotação de `ENCRYPTION_KEY` antes de produção continua pendente.
