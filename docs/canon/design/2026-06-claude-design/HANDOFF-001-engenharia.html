Sistema de status

um StatusBadge, seis tons

Tabela definitiva. Cor sempre acompanhada de texto e ponto, nunca cor sozinha. O componente é único (StatusBadge); o que muda é o tone. A “aprovação” tem tratamento próprio — borda tracejada e ponto pulsante — para se destacar como ação humana pendente.

TomToken de corExemploEstados que cobre
neutralmuted / borda borderDisponívelDisponível · Rascunho · Em triagem · Pausado · Encerrado · Não configurado
infoaccent + accent-softEm andamentoEm andamento · Em teste · Em análise · Aguardando cliente · Acionamento em andamento
successsuccessConectadoConectado · Ativo · Pronto para acionamento · Resolvido
warningwarningPrecisa configurarPrecisa configurar · Aguardando documento · Aguardando seguradora · Aguardando prestador · Atrasado
dangerdangerCom erroCom erro · Bloqueado · Cancelado · Handoff humano
approvalaccent · borda tracejada + pulsoAguardando aprovaçãoAguardando aprovação (HITL) — sempre que uma ação externa espera decisão humana
StatusBadge — assinatura
type StatusTone =
  'neutral'|'info'|'success'|
  'warning'|'danger'|'approval';

<StatusBadge tone="approval"
  label="Aguardando aprovação" />
regras de uso
// 1. nunca só cor — sempre ponto + texto
// 2. approval = borda dashed + dot pulsante
// 3. mapear status do ADR-003 → tone
//    via util statusToTone(status)
// 4. handoff_humano = danger (atenção máxima)
// 5. aguardando_aprovacao = approval

05

Ordem de implementação

batches pequenos · fechados · testáveis

Sequência alinhada ao ROADMAP. Cada batch é fechado e tem um critério de pronto. Não pular ordem: a fundação (tokens + shell) sustenta tudo. Cada tarefa entregue ao Claude Code deve referenciar a ficha §4 correspondente.

B0

Fundação visual

Instalar Geist, aplicar globals.css (:root) e tailwind.config.js do §1. Criar primitivos: Button, Tile, StatusBadge, EmptyState, Skeleton, ErrorState.

PRONTO: Storybook/rota de sandbox mostra tokens e primitivos com os hex corretos. typecheck + build verdes.
B1

Shell de navegação

AppSidebar (4 pilares + recents + user), BottomNav mobile, Breadcrumb/BackLink, PageHeader. Layout responsivo.

PRONTO: navega entre os 4 pilares; ativo/hover corretos; mobile mostra bottom-nav.
B2

Chat-first em /dashboard

ChatEmptyState + ChatComposer + 2 ShortcutChip. Ligar ao chat/stream existente do runtime, sem alterar backend.

PRONTO: empty state aprovado renderiza; envio funciona; histórico aparece em “Recentes”.
B3

Padrões-mestre como componentes

GalleryGrid/Card, CategoryFilter, SearchField, DetailHeader, Tabs/SegmentedControl, SideCard, PermissionModal/BottomSheet, RiskBar. Tudo mockado em sandbox.

PRONTO: os três padrões A/B/C renderizam isolados, desktop e mobile.
B4

Shells dos módulos (sem backend novo)

Auxiliares (lista/galeria), Conectores (galeria+detalhe+modal), Seguradoras (lista+página), Atendimentos (lista+caso). Dados mockados.

PRONTO: todas as rotas do §4 abrem com layout aprovado e estados vazio/loading/erro.
B5

Auxiliar de Resumo funcional

Detalhe + Stepper de ativação + execução manual real sobre conversa selecionada → resumo + run salvo (custo, histórico). Sem envio externo.

PRONTO: rodar o Auxiliar gera resumo real e grava execução visível em Execuções.
B6

Atendimento base + HITL

Conectar lista/detalhe de Caso a dados reais; sistema de status do ADR-003; HitlBanner com estado “aguardando aprovação” antes de qualquer ação externa.

PRONTO: caso real exibe status correto; ação externa fica bloqueada até “Aprovar”.

Regra de ouro do batch: um PR = uma ficha §4 ou um grupo de componentes §2. Nunca “melhore o módulo X”. Sempre escopo fechado + critério de pronto + fora de escopo explícito.

→

Como transformar isto no material perfeito para o Claude Code

Sem atropelar. Passo a passo, até termos tudo pronto para entregar. Este handoff é o documento-mãe; cada passo abaixo gera os artefatos finais que o Claude Code recebe.

Aprovar este handoff como canônico

Você revisa §1–§5. Aprovado, ele vira HANDOFF-001 em docs/canon/ — a referência única. Qualquer dúvida de implementação volta aqui, não ao “achismo”.

Congelar os tokens no repositório primeiro

O Batch B0 entra sozinho: globals.css + tailwind.config.js + primitivos. Nada mais avança antes dos tokens estarem no código. É a base que impede divergência visual.

Gerar um card de tarefa por ficha §4

Cada tela do §4 vira uma tarefa fechada para o Claude Code, copiando os campos: rota, layout, componentes, estados, shell×dado, fora de escopo, HITL. Eu posso escrever esses cards prontos quando você pedir.

Seguir a ordem dos batches, um PR por vez

B0 → B1 → … → B6. Cada PR referencia a ficha e o critério de PRONTO. O Claude Code executa dentro do escopo; não decide produto. Roda typecheck + build antes de fechar.

Validar cada batch contra o mockup

Ao fim de cada PR, comparar a tela renderizada com a Leva correspondente (1, 2 ou 3). Diferença visual = bug, não “interpretação”. Ajustar antes de seguir.

Só então ligar dados reais

Shells (B4) antes de dados (B5/B6). Primeiro a UX aprovada com mock; depois o runtime. Ações externas só entram com o estado HITL implementado e visível.

O que eu (Design) posso entregar a seguir, quando você aprovar este handoff: (1) os cards de tarefa prontos, um por ficha §4, no formato que o Claude Code executa; (2) os ícones do produto como set SVG consistente; (3) a planilha de mapeamento status do ADR-003 → StatusTone. É só dizer qual começar — seguimos passo a passo, sem pular etapa.

HANDOFF-001 · pronto para revisão

Este documento fecha a ponte design → engenharia. Aprovado, vira canônico e destrava o Batch B0. A partir daí, cada tela aprovada nas Levas 1–3 tem tokens, componentes, status, spec e ordem — tudo que o Claude Code precisa para implementar sem reinterpretar uma única decisão visual.
