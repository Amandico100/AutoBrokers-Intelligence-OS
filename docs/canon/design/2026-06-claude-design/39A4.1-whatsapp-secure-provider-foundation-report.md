# 39A4.1 — WhatsApp Secure Provider Foundation Report

> **Status:** concluído, **commitado e pushado** · py_compile/typecheck/build verdes · **sem SQL/migration/schema** · **sem envio real, sem QR, sem n8n/Evolution**.
> **Data:** 2026-06-08 · **Modelo:** Claude Opus 4.8 · **Branch:** main

## 1. Resumo executivo
Fundação segura de WhatsApp (Caminho C): criada a **camada de provider** (`WhatsAppProvider` + `ZApiProvider`) e os **helpers de criptografia de segredos de integração** (reuso do `EncryptionService`). O `integration_service` passa a **descriptografar `token`/`client_token` em memória** ao entregar a integração ao runtime; o `whatsapp_service` virou um **shim que delega ao `ZApiProvider`** e **não loga mais a URL com token** (P0 corrigido). A rota admin de integrações **bloqueia gravação de token/client_token em texto puro** até o fluxo seguro. Nada é enviado de verdade; DRY_RUN preservado.

## 2. Arquivos alterados
**Criados (backend):**
- `backend/app/services/whatsapp/__init__.py`, `types.py` (`SendResult`), `provider.py` (`WhatsAppProvider`), `zapi_provider.py` (`ZApiProvider`), `integration_secrets.py` (encrypt/decrypt + prepare for storage/runtime).
**Alterados (backend):**
- `backend/app/services/integration_service.py` — `get_integration_by_phone` e `get_whatsapp_integration` retornam integração com segredos **descriptografados em memória** (`prepare_integration_for_runtime`); log de `identifier` mascarado.
- `backend/app/services/whatsapp_service.py` — shim que **delega ao `ZApiProvider`**; removido o log de URL com token.
**Alterado (Next):**
- `app/api/admin/integrations/route.ts` — **rejeita** `token`/`client_token` no payload (400) — impede plaintext até o secret flow.
**Docs:** este relatório.

## 3. Padrão de criptografia encontrado
`backend/app/services/encryption_service.py` — `EncryptionService` (Fernet/AES, chave `ENCRYPTION_KEY`), singleton `get_encryption_service()`. Já usado para **MCP tokens** (`mcp_oauth_service`: `encrypt(access_token)`/`decrypt(...)`) e **LLM/vision keys** (`agent_config.py`). Reaproveitei **esse mesmo padrão** para `integrations.token`/`client_token` (sem inventar cripto nova, sem duplicar em TS).

## 4. Como tokens de integrations passam a ser tratados
- **Runtime (leitura):** `integration_service` chama `prepare_integration_for_runtime(row)` → `decrypt_integration_secret` em `token` e `client_token` (em memória, nunca logado).
- **Storage (gravação):** `prepare_integration_for_storage(data)` → `encrypt_integration_secret` (pronto para o secret flow do 39A4.2 gravar cifrado).
- **Compatibilidade:** se um valor **não** for ciphertext válido (legado plaintext), `decrypt_integration_secret` faz fallback tratando-o como plaintext, com **warning sem o valor**. Como o banco está **sem integrações hoje**, o caminho normal é tudo cifrado daqui para frente.

## 5. Como o provider adapter foi criado
`WhatsAppProvider` (ABC) define `send_text/send_audio/send_image -> SendResult` + `validate_config`. `SendResult` é estruturado: `success`, `provider`, `dry_run`, `message_id?`, `error?` (nunca contém segredo). EvolutionProvider fica como **fase futura** atrás da mesma interface (nota no código).

## 6. Como Z-API foi encapsulado
`ZApiProvider` centraliza a chamada Z-API REST (`send-text/send-audio/send-image`), monta a URL `{base_url}/{instance_id}/token/{token}/{endpoint}` **sem logá-la**, adiciona `Client-Token` no header quando houver (sem logar), e retorna `SendResult`. `WhatsappService.send_message/audio/image` agora **delegam** a `get_zapi_provider()` mantendo o contrato (bool; `send_message` levanta exceção em falha, como antes) usado por `webhook.py`.

## 7. Como DRY_RUN foi preservado
`ZApiProvider._post` checa `settings.DRY_RUN` **antes** do HTTP e retorna `SendResult(success=True, dry_run=True)` simulando o envio (igual ao comportamento anterior do `whatsapp_service`). O DRY_RUN do `integration_service.get_integration_by_phone` (que mascara token/instance) permanece intacto.

## 8. Como logs foram endurecidos
- **Removido** o `logger.debug("[WHATSAPP] URL: {url}")` (vazava token na URL).
- Erros HTTP logam **apenas o status** (`http_429`), **não** `response.text`.
- Erros de request logam `type(e).__name__`, sem URL/credencial.
- **Telefone sempre mascarado** (`...1234`); `Client-Token`/token nunca logados.
- `integration_service` agora **mascara o `identifier`** no log de sucesso (antes logava o telefone conectado inteiro).

## 9. O que foi feito no admin integrations
`POST /api/admin/integrations` agora **rejeita (400)** se `token` ou `client_token` vierem no payload, com mensagem clara ("…configuradas pelo fluxo seguro de conexão. Esta rota não aceita token/client_token."). Isso **impede gravação de segredo em texto puro** por essa rota legada até o secret flow seguro (39A4.2). Não duplicamos criptografia em TS; a UI admin não foi reescrita (apenas o write inseguro foi bloqueado).

## 10. O que NÃO foi implementado
Envio real, conexão real, QR Code, Evolution, n8n, UI (tenant/admin nova), HITL runtime de envio, Auxiliar de Cobrança, migração de dados (não há tokens para migrar). Sem SQL/migration/schema. `EvolutionProvider` apenas como nota futura.

## 12. Riscos/remanescentes
- **Admin write bloqueado:** o cadastro de WhatsApp por `/admin/integrations` fica indisponível até o **secret flow (39A4.2)** — intencional (impede plaintext). A UI admin ainda mostra campos de token, mas o POST recusa; documentar/ocultar em batch próprio.
- **Compat plaintext:** mantida só por segurança de ambientes legados (com warning, sem valor). Banco atual está vazio → caminho cifrado é o padrão.
- **DRY_RUN ainda é global** (env), não por conexão; flag por `tenant_connections.metadata` é evolução (39A4.2/3).
- **Webhook inbound** segue sem assinatura HMAC (P1, fora deste escopo).
- Validação real de roundtrip encrypt/decrypt depende de `ENCRYPTION_KEY` no ambiente; não forçada aqui (funções puras, testadas por inspeção).
