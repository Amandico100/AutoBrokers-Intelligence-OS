> **Data:** 2026-06-09 · **Modelo:** Claude Opus 4.8 · **Branch:** main

## 1. Resumo
Fundação de runtime para conhecimento **global** sobre o RAG existente do Smith (SPEC-003), **sem ativar nada por padrão**. Criado um módulo de helpers de escopo; os documentos passam a carregar **campos de conhecimento** (scope/knowledge_class/…) no payload do Qdrant; o `search_similar` aceita filtros de **scope/curadoria**; e há uma **interface `include_global` (default False)** em `search_service`/`langchain_service` que, quando ligada (batch futuro), busca uma coleção global dedicada e faz merge. Nada de global é consultado hoje (nenhum caller passa `include_global=True`). Tenant/agent retrieval segue idêntico ao 41C.1.

## 3. Como o scope é normalizado
`knowledge_scope.normalize_document_scope(agent_id, requested_scope=None)`: usa `requested_scope` **só se válido**; senão **`agent`** quando há `agent_id`, **`tenant`** caso contrário. Escopos canônicos (SPEC-003): `tenant`, `agent`, `global_autobrokers`, `global_carrier`, `workflow`, `connector`. **Compatibilidade:** documentos antigos sem `scope` → tratados como `tenant` (default em retrieval e no payload). Upload atual (filesystem) já grava `scope` no row.

## 4. Como o payload Qdrant foi preparado
`extract_payload_extras(metadata, agent_id)` reúne, de forma tolerante, os campos de conhecimento presentes (`scope`, `knowledge_class`, `namespace`, `version`, `curation_status`, `visibility`, `valid_until`, `source_hash`, `carrier_slug`, `product_slug`) — sempre com `scope` definido. `insert_embeddings` recebe `knowledge_extras` e os mescla no **payload raiz** de cada chunk (sem sobrescrever campos base; **nunca** conteúdo/segredo/PII). Uploads atuais passam a marcar `scope` no Qdrant via `ingestion_service`.

## 5. Como o retrieval global fica desativado por padrão
- `include_global` tem **default `False`** em `smart_search`, `_execute_search`, `search_documents`, `get_rag_context`. **Nenhum caller passa `True`** (confirmado por scan) → global é **dormante**.
- Quando ligado (futuro), `build_global_search_kwargs()` busca a coleção dedicada **`autobrokers_global`** com `scope_match=[global_autobrokers, global_carrier]` + `curation_published_only=True`, sem `agent_id`. Como essa coleção **ainda não existe**, `search_similar` retorna `[]` com segurança (checa `collection_exists`).
- `merge_rag_results` une tenant/agent + global, **dedupe** por `(document_id, chunk_index)` mantendo o maior score; o reranker reordena.

## 6. Como agent/tenant retrieval segue funcionando
Inalterado vs 41C.1: caminho vivo (`knowledge_base` tool → `SearchService` → `qdrant.search_similar(agent_id, include_tenant_wide=True)`) → agente vê **seus docs + tenant-wide**, **nunca de outro agente**. `scope_match`/`curation` só entram quando explicitamente passados (global). Sem `agent_id` → company-wide (compat).
