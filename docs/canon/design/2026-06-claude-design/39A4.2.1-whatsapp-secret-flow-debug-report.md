## 1. Causa raiz encontrada
A mensagem **"Falha ao configurar a conexão."** é o **fallback genérico** da rota Next, usado **somente quando o backend responde algo que não é JSON com `detail`/`error`**. Um erro real do FastAPI (validação, 401, 404, 500 tratado) **sempre** carrega `detail` — e a rota mostraria esse texto. Como o usuário viu o genérico, o backend retornou um **corpo não-JSON** (página de erro de gateway/HTML — 502/503, ou rota não servida) para `POST /api/whatsapp-integrations/configure`.

Diagnóstico de código (tudo verificado, **sem** bug de registro):
- `main.py` registra o router corretamente (módulo, linhas 166–169): `prefix=/api/whatsapp-integrations`.
- Imports do backend são válidos: `AsyncSupabaseClient`/`get_async_db` existem em `core/database.py` (214/385) e os Auxiliares usam exatamente os mesmos. **Sem import circular** (`services/__init__.py` importa `encryption_service` na linha 7, **antes** de `integration_service` na linha 9).
- `py_compile` valida **apenas sintaxe** — não detecta rota ausente no servidor.

**Conclusão:** o serviço de **API não está servindo a rota nova** — quase certamente o **deploy da API não reconstruiu/reiniciou** com o código do 39A4.2 (ou o container está em erro/restart no EasyPanel, devolvendo HTML). A rota Next mascarava o status real.

> Observação: a outra possibilidade (envio/cripto/upsert quebrando no backend) **não** produz o genérico — produziria um `detail`. Mesmo assim, endurecemos o backend por robustez.

## 3. Correção aplicada
**Diagnóstico (o que resolve o "erro mudo"):**
- As rotas Next agora leem o corpo bruto e retornam: o `detail`/`error` do backend **ou** `"Backend retornou <status> em /api/whatsapp-integrations/configure — resposta do backend: <trecho>"`. O `catch` de rede passa a incluir a mensagem (`ECONNREFUSED`, `getaddrinfo`, etc.). Status 502 quando o corpo é inesperado. **Nenhum segredo é exposto** (o corpo dessas rotas nunca contém token).
- **`GET /api/vault/whatsapp/health`** (logado) chama `GET /api/whatsapp-integrations/health` (chave interna) e retorna `{ reachable, status, backend_host, body }`. Permite confirmar em 1 request se o Web alcança a rota do backend e se a criptografia está configurada.

**Robustez (backend):**
- Criptografia em bloco próprio: se `EncryptionService` falhar → `500 "Encryption unavailable (<Tipo>)"` (sem valor).
- Gravação por **select→update/insert** em `integrations` (procura por `provider+identifier`), evitando depender de `upsert(on_conflict=...)` no client assíncrono. Erros retornam `500 "Failed to store integration (<Tipo>)"`.
- `GET /health` (chave interna) → `{ success, service, encryption_configured }` (encripta um literal de teste — **não** um segredo).

1. **Pré-checagem de reachability:** abrir no navegador (logado)
   `https://<web-host>/api/vault/whatsapp/health`
