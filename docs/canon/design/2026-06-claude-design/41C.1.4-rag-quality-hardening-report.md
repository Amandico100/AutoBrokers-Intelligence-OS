> **Data:** 2026-06-11 · **Modelo:** Claude Opus 4.8 · **Branch:** main

## 1. Causa estrutural
O reranker **Cohere** (`rerank_service.rerank`) faz **pass-through** quando não há `COHERE_API_KEY` (`return docs[:top_k]`, sem injetar `rerank_score`) ou quando a API falha. Nesses casos, `smart_search` lia `results[0].get("rerank_score", 0)` → **0** → abaixo do `THRESH_MIN (0.40)` → resultados corretos eram descartados (corrigido como sintoma no 41C.1.3 via lexical rescue). Este batch trata a raiz da **observabilidade/score**, sem reescrever o serviço.

## 2. `rerank_score` × score bruto do Qdrant
- **`rerank_score` (Cohere `rerank-multilingual-v3.0`):** relevância cross-encoder normalizada **0..1** — comparável ao `THRESH_MIN=0.40`.
- **Score bruto do Qdrant:** na busca **híbrida** usamos **RRF Fusion** (`Fusion.RRF`), cujo score é de **rank-fusion** (magnitude pequena, ~0.01–0.03), **não** comparável à escala do threshold do Cohere. Na busca dense-only seria cosseno (~0..1).
- **Implicação honesta:** usar o score do Qdrant como fallback **restaura observabilidade** e elimina o "zero falso", mas para busca híbrida ele tende a ficar abaixo de 0.40. Por isso o **lexical rescue (41C.1.3) continua sendo a camada operativa** quando o reranker está ausente. O fallback evita decisões erradas baseadas em `0` e dá visibilidade (`score_source`), sem inventar relevância.

## 3. O que foi alterado
- `backend/app/services/rerank_service.py`: `is_available()` e `health()` — status leve **sem expor chave e sem chamada à API** (`configured/provider/status/safe_to_run`). `safe_to_run=True` sempre (há fallback + rescue).
- `backend/app/services/search_service.py`:
  - Helpers puros `_get_effective_score(result)` (rerank → qdrant → 0) e `_get_score_source(result)` (`rerank`|`qdrant`|`fallback_zero`).

  - `_format_response` / `_build_chunks_metadata`: usam score efetivo; gravam `score_source` e `reranker_available` por chunk; resposta carrega `effective_score`, `score_source`, `reranker_available`.
  - Dicts de `found=False` agora também trazem `effective_score`, `score_source`, `reranker_available`.
- `backend/app/api/documents.py`:
  - `GET /rag-health`: novo bloco `reranker` (`configured/provider/status/safe_to_run`).
  - `POST /rag-debug`: agora retorna `effective_score`, `score_source`, `reranker_available`, `reranker` (resumo), `used_in_context_count`, e por chunk `score_source`. Sem documento integral.

## 4. Ordem de decisão (FASE 5 preservada)
1. calcula **score efetivo** (rerank → qdrant → 0);
2. se passar o threshold → usa normalmente (`hybrid_direct`/HyDE);
3. se abaixo do threshold → tenta **lexical rescue** (41C.1.3);
4. se o rescue falhar → `found=false`.

## 5. Comportamento com / sem `COHERE_API_KEY`
- **Com chave (status `ok`, provider `cohere`):** rerank real injeta `rerank_score` 0..1 → `score_source=rerank`; threshold e qualidade fina funcionam como projetado.
- **Sem chave (status `missing_key`, provider `none`):** rerank é pass-through → sem `rerank_score` → `score_source=qdrant` (RRF) → normalmente abaixo do threshold → **lexical rescue** entra e o RAG responde. Logs mostram "Reranker bypass detected; using Qdrant score fallback". `safe_to_run=True`.
- **Chave presente mas client falhou (status `error`, provider `fallback`):** idêntico ao bypass em runtime; fallback + rescue cobrem.

## 6. Por que continua seguro

## 8. Recomenda configurar `COHERE_API_KEY` agora?
**Recomendado, mas não bloqueante.** Sem ela o RAG funciona (fallback + rescue), porém a **precisão de relevância** fica degradada (sem reordenação cross-encoder; dependente do rescue lexical). Para produção, configurar `COHERE_API_KEY` melhora ranking e reduz dependência do rescue. **Não** foi feito neste batch (envolve segredo/infra). Sugestão: setar a chave no ambiente da API e revalidar via `/rag-health` (`status:"ok"`).
