# 41C.1.5 — Local RAG Acceptance & Regression Checklist

> **Tipo:** trava de qualidade (aceitação/regressão) do **RAG local do Smith** antes do conhecimento global.
> **Data:** 2026-06-11 · **Modelo:** Claude Opus 4.8 · **Branch:** main
> **Pré-condição:** batches 41C.1, 41C.1.1, 41C.1.2, 41C.1.3, 41C.1.4 aplicados e deployados (API).

## 1. Objetivo
Garantir que qualquer dev/IA futura consiga **validar o RAG local** e detectar regressão **antes** de avançar para Global Knowledge (41C.2C). Este documento é o **gate**: se algum teste reprovar, **não** avançar para global.

## 2. Estado esperado do sistema
- Upload local (Admin → Base de Conhecimento) usa o **RAG do Smith existente** (sem motor paralelo).
- Documento persiste em `public.documents`; arquivo no **MinIO** (bucket `documents`); chunks no **Qdrant** `company_{company_id}`.
- Payload do chunk no Qdrant contém **`agent_id`** (filtro) + `document_id`, `chunk_index`, `content`, `metadata`, e campos de escopo (`scope`, …). **`company_id` é isolado pelo nome da coleção** (`company_{company_id}`), não pelo payload.
- Retrieval do chat: `SearchService.smart_search(company_id, query, agent_id, is_hyde_enabled, include_global=False)` → `qdrant.search_similar(agent_id, include_tenant_wide=True)`.
- **`include_global` = OFF** em todos os callers (default `False`; nenhum passa `True`).
- Reranker Cohere é **opcional**: sem chave → fallback de score + lexical rescue (41C.1.3/41C.1.4).

## 3. Pré-requisitos de ambiente
- API up; `OPENAI_API_KEY` válido (embeddings `text-embedding-3-small`).
- MinIO e Qdrant acessíveis pela API.
- (Opcional) `COHERE_API_KEY` para reranking fino — **não obrigatório**.
- Acesso master/admin (cookie `smith_admin_session`) para os probes.

## 4. Health checks obrigatórios
`GET /api/admin/proxy/documents/rag-health` deve retornar:
```
documents_route: true
minio: true
qdrant: true
reranker: { configured, provider, status, safe_to_run:true }
include_global_default: false
minio_bucket: "documents"
qdrant_collection_pattern: "company_{company_id}"
```
**Reprova** se `minio` ou `qdrant` = false, ou `include_global_default` ≠ false, ou `safe_to_run` ≠ true.

## 5. Teste A — RAG agent-specific
1. Criar um TXT de teste **genérico** (sem PII, sem dado real de cliente) com uma frase-âncora única (ex.: um código fictício gerado no momento — **não** versionar o código aqui).
2. Admin → Empresa de teste → Base de Conhecimento → agente **A** → Modo RAG Semântico → Upload.
3. Esperado: documento aparece na lista → `processing` → **`completed`**, `chunks_count ≥ 1`, visível em "Ver Chunks".
4. `POST /api/admin/proxy/documents/rag-debug` `{company_id, agent_id:A, query: <pergunta sobre a frase-âncora>}` → `found=true`, `document_ids` contém o doc, `used_in_context_count ≥ 1`.

## 6. Teste B — RAG tenant-wide
1. Subir um doc com **`agent_id` nulo** (tenant-wide) na mesma empresa (se a UI permitir; caso contrário validar via diagnóstico SQL que existe doc com `agent_id IS NULL` e `scope='tenant'`).
2. `rag-debug` com `agent_id:A` deve **recuperar** o doc tenant-wide (via `include_tenant_wide=True`).
3. **Aprova** se o agente A vê: docs próprios **+** tenant-wide.

## 7. Teste C — isolamento agent A vs agent B
1. Doc exclusivo do agente **A**.
2. `rag-debug` com `agent_id:B` (mesmo company) e a mesma query → **NÃO** deve retornar o doc de A (`document_ids` sem o doc de A; idealmente `found=false` ou só tenant-wide).
3. **Reprova** imediatamente se B vê doc de A (vazamento entre agentes).

## 8. Teste D — chat retrieval
1. No chat do agente **A**, perguntar sobre a frase-âncora.
2. Esperado: resposta usa o conteúdo recuperado (não "não encontrei na base").
3. Logs da API: `[RAG Prefetch] found=true …` e/ou `[Search] … strategy=…`.

## 9. Teste E — logs / conversation_logs
- `conversation_logs` da pergunta: `rag_chunks_count ≥ 1`, `search_strategy` preenchido, chunk com `used_in_context=true` e `score_source` ∈ {`rerank`,`qdrant`} (**não** `fallback_zero`).
- **Reprova** se `rag_chunks_count=1` mas `used_in_context=false` **e** a resposta nega ter achado (regressão do tipo 41C.1.2/41C.1.3).

## 10. Teste F — reranker SEM `COHERE_API_KEY`
- `rag-health.reranker.status` = `missing_key` (provider `none`), `safe_to_run=true`.
- Chat ainda recupera (fallback de score + lexical rescue). `score_source` = `qdrant` (ou rescue ativo). **Aprova** mesmo sem Cohere.

## 11. Teste G — reranker COM `COHERE_API_KEY` (futuro)
- `rag-health.reranker.status` = `ok` (provider `cohere`).
- `rag-debug`/logs: `score_source=rerank`, `max_score` na escala 0..1. **Aprova** se relevância melhora sem quebrar isolamento.

## 12. Critérios de aprovação
- Health: minio/qdrant true, `include_global_default=false`, reranker `safe_to_run=true`.
- A, B, D, E passam; **C passa obrigatoriamente** (sem vazamento).
- F passa sem Cohere.

## 13. Critérios de reprovação
- Qualquer vazamento agent A↔B (Teste C).
- `include_global` ligado sem SPEC/aprovação.
- Documento `completed` mas chat nega achar com `rag_chunks_count≥1`.
- `minio`/`qdrant` false.
- Aparição de `scope` global (`global_autobrokers`/`global_carrier`) **antes** da ativação oficial.

## 14. O que NÃO mexer
MinIO/Qdrant config, embeddings (`text-embedding-3-small`), chunking default, threshold global (sem SPEC), lexical rescue (41C.1.3), fallback de score (41C.1.4), filtro `agent_id`, `include_tenant_wide=True`, `include_global` default. Sem schema/SQL DDL/migration/nova coleção/nova tabela.

## 15. Como interpretar documentos "processing"
`processing` = ingestão em andamento (extração → chunk → embeddings → Qdrant). É **transitório**. Só vira `completed` (com `chunks_count`) ou `failed` (com `error_message`).

## 16. Quando aguardar e quando considerar falha
- **Aguardar** até ~1–2 min para TXT pequeno. Recarregar a lista.
- **Falha** se: ficar `processing` indefinidamente (> alguns minutos para arquivo pequeno) **ou** `failed` com `error_message`. Verificar `rag-health` (minio/qdrant) e logs da API (`[Search]`, ingestion). Não reprocessar em loop.

## 17. Como testar sem conteúdo sensível
- Usar **arquivo fictício**, sem PII, sem dado real de corretora/cliente.
- Frase-âncora deve ser um código/termo **inventado na hora** — **não** comitar o termo em docs/SQL/código.
- Probes retornam apenas previews ≤160 chars, ids, score, strategy — nunca documento integral.

## 18. Como limpar documentos de teste
- Admin → Base de Conhecimento → excluir o documento de teste (remove DB + MinIO + Qdrant via `delete_document`).
- Confirmar via `rag-debug` que a query de teste não retorna mais o doc.

## 19. Próximo passo: Global Knowledge
Só avançar para **41C.2C** (coleção `autobrokers_global` + ingestão curada) **após** todos os critérios de §12 passarem e nenhum doc com `scope` global existir antes da ativação (§13). Global permanece **OFF** até o batch oficial.
