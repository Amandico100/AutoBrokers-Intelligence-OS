## 3. Segurança Next → Backend
O endpoint FastAPI fica acessível publicamente na API (EasyPanel). Antes, confiava só no `company_id` do body. Agora exige um **segredo interno** que apenas o Next conhece (env server-only). O Next continua sendo a fronteira de auth (iron-session + `users_v2`, `company_id` derivado no servidor), e o backend revalida o escopo (tenant auxiliary + conversa por `company_id`). Resultado: chamada direta externa ao endpoint é rejeitada antes de qualquer processamento.

## 4. Header interno implementado
- **Header:** `X-AutoBrokers-Internal-Key`.
- **Backend** (`_require_internal_key`): `expected = BACKEND_INTERNAL_API_KEY or ADMIN_API_KEY`. Sem env → **HTTP 500** `"Internal API key not configured"` (não processa). Header ausente/incorreto → **HTTP 401** `"Unauthorized internal request"`. Comparação com `hmac.compare_digest` (constante no tempo). Validação é a **primeira** coisa do endpoint, antes de tocar no banco.
- **Next:** envia o header em toda chamada ao backend; valor nunca vai ao client.

## 5. Variáveis usadas
- `BACKEND_INTERNAL_API_KEY` (preferida, futura) **ou** `ADMIN_API_KEY` (fallback — já existe no EasyPanel).
- Lidas server-side: Next `process.env.*` (sem `NEXT_PUBLIC`), Backend `os.getenv(...)`.
- **Não** criei `.env`, **não** imprimo o valor, **não** exponho ao client, **não** coloquei valor real em arquivo. Para o deploy, basta que `ADMIN_API_KEY` exista no **Next** e no **Backend** (já existe no backend; confirmar no serviço Web).

## 6. Seleção de conversa recente
`_pick_recent_conversation_with_messages` agora usa `_list_recent_conversations`, robusto a diferenças de schema:
1. `select id, updated_at, created_at` ordenado por `updated_at desc`;
2. fallback `select id, created_at` ordenado por `created_at desc`;
3. fallback final `select id` (sem ordenação), tudo filtrado por `company_id`.
Cada tier captura exceção e **loga o fallback** (apenas `type(e).__name__`, sem vazar dados). Mantida a verificação de mensagens (retorna a 1ª conversa com mensagens). **Não depende mais de `last_message_at`.**

## 7. Persistência em auxiliary_runs
- Query do tenant auxiliary: `select id, template_id, company_id, slug, status`.
- `_create_run(company_id, tenant_auxiliary_id, template_id, conversation_id, run_input, user_id)` grava colunas próprias:
  `company_id, tenant_auxiliary_id, template_id, conversation_id, user_id, run_type, status, input, started_at, metadata`.
- `template_id`/`user_id`/`conversation_id` aceitam `null`. `conversation_id` continua também em `input` (e em `metadata` no sucesso) para conveniência, mas a **coluna própria** é preenchida.

## 10. Como testar depois do deploy
Pré-requisitos: `ADMIN_API_KEY` (ou `BACKEND_INTERNAL_API_KEY`) configurada **no Web (Next) e no Backend** com o **mesmo valor**; backend no ar.
1. **Via app (logado):** no DevTools →
   ```js
   await fetch('/api/auxiliaries/resumo-atendimentos/run', { method:'POST',
     headers:{'Content-Type':'application/json'}, body: JSON.stringify({}) }).then(r=>r.json())
   ```
   Esperado: `{ success:true, run:{ id, status:'succeeded', output:{...} } }`.
2. **Hardening (negativo):** chamar o backend **direto** (sem header) deve responder **401** `Unauthorized internal request`; com env ausente, **500** `Internal API key not configured`.
3. **Persistência:** conferir no Supabase que a nova linha de `auxiliary_runs` tem `template_id`, `conversation_id`, `user_id` preenchidos (quando disponíveis), além de `output`/`token_usage`/`cost_usd`.
4. **Seleção automática:** rodar sem `conversation_id` deve escolher uma conversa com mensagens mesmo que `last_message_at` não exista.

## 11. Riscos/remanescentes
- **Chave compartilhada Web↔Backend:** se `ADMIN_API_KEY` divergir entre os serviços, o run retorna 401. Garantir o mesmo valor nos dois no EasyPanel. Migrar para `BACKEND_INTERNAL_API_KEY` dedicada é o passo limpo futuro.
- **Contrato de colunas:** o insert assume que `auxiliary_runs` tem `template_id`/`conversation_id`/`user_id` e `tenant_auxiliaries` tem `template_id` (confirmado pelo fundador). Se algum nome divergir, é ajuste de 1 linha; leituras usam `select('*')`.
- **Sem débito de crédito** ainda (apenas log em `token_usage_logs`). **Sem RAG**, **síncrono**, **read-only**.
