# 41C.1.2 — RAG Chat Retrieval Fix Report

> **Status:** concluído · py_compile verde · `git diff --check` limpo · **sem motor paralelo, sem schema/SQL, sem mexer em MinIO/Qdrant/upload, sem ativar global, sem hardcode de palavra-chave**.
> **Data:** 2026-06-11 · **Modelo:** Claude Opus 4.8 · **Branch:** main

## 1. Causa raiz
O RAG do chat era **somente uma tool** (`knowledge_base_search`). O LLM decide se chama a tool; quando ele **não emite o tool_call**, o Tool Node nunca roda, nenhum chunk entra no contexto e o agente responde *"Eu ainda não encontrei essa informação na base da corretora"* — mesmo com o documento `completed` e o chunk visível. Sintoma confirmado: `conversation_logs.rag_chunks_count = 0` (a busca nunca aconteceu).

## 2. Correção (prefetch determinístico)
Adicionado um **prefetch de RAG determinístico** dentro de `_build_initial_state` (graph), **antes** de montar `composite_prompt`/`initial_state`. Ele usa **o mesmo** `SearchService.smart_search` do caminho vivo (não cria motor novo) e injeta o conteúdo recuperado no **contexto dinâmico** (não cacheado) do prompt. É **aditivo**: a tool `knowledge_base_search` continua ativa (o LLM ainda pode buscar de novo).

### Fluxo novo
1. `should_prefetch_rag(user_message)` decide se vale buscar (pula saudações/curtas; dispara em palavras de intenção de conhecimento ou mensagens > 25 chars).
2. Se sim → `smart_search(company_id, user_message, agent_id, is_hyde_enabled, include_global=False)` via `asyncio.to_thread` (não bloqueia o event loop).
3. Se `found=True` e há `content` → injeta no `dynamic_context`:
   `=== 📚 CONTEXTO RECUPERADO DA BASE DE CONHECIMENTO === … === FIM === ` + instruções ("Se a resposta estiver no contexto recuperado, responda com base nele; não diga que não encontrou; use com precisão; não invente fora do contexto interno").
4. Popula `initial_state`: `rag_context`, `rag_chunks`, `rag_search_time_ms`, `search_strategy`, `retrieval_score`.

Como o **Agent Node** monta o system message a partir de `static_prompt` + `dynamic_context` (Anthropic, com cache) ou de `system_prompt`/`composite_prompt` (demais), a injeção chega ao LLM nos dois caminhos. O `static_prompt` (cacheável) **não** é tocado — o contexto recuperado entra só no trecho **dinâmico** (correto, é por-pergunta).

## 3. Isolamento e segurança preservados
- `agent_id` repassado ao `smart_search` → **isolamento por agente** intacto (41C.1); `include_tenant_wide=True` continua no caminho interno do `search_service` (inalterado).
- `include_global=False` em **todas** as chamadas → **global segue dormante** (41C.2B/SPEC-003).
- Sem hardcode de palavra-chave/`company_id`/`agent_id`. Sem MinIO/Qdrant/upload/schema/SQL.
- Logs **seguros** (sem conteúdo/segredo/PII): `[RAG Prefetch] skipped: greeting/short`, `[RAG Prefetch] searching company=… agent=…`, `[RAG Prefetch] found=true chunks=N score=X strategy=Y`, `[RAG Prefetch] no result`. Erro do prefetch é **engolido** (`type(e).__name__`) — nunca quebra o chat.

## 4. Logging (conversation_logs)
- `rag_chunks` / `rag_chunks_count` agora vêm do prefetch quando o LLM não chama a tool → **deixa de ser 0** quando há recuperação. Tool Node e Agent Node **preservam** `rag_chunks` do state (não sobrescrevem), então o prefetch sobrevive até o log_node.
- `log_node` ganhou **fallback**: se não houve tool_message, usa `search_strategy`/`retrieval_score` do prefetch (state). Sem fallback, ficaria `None` — não quebrava, mas agora o log fica completo.

## 5. Diagnóstico master-only
Novo `POST /api/documents/rag-debug` (`require_master_admin`, via admin proxy). Input: `company_id, agent_id, query`. Output: `found, max_score, strategy, chunks_count, search_time_ms, document_ids, agent_ids, content_preview` (≤160 chars/chunk). **Sem segredo, sem documento integral.** Executa o mesmo `smart_search` do chat → permite confirmar retrieval sem ler logs.
