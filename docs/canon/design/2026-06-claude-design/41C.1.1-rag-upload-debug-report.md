# 41C.1.1 — Local RAG Upload Pipeline Debug Report

> **Status:** concluído, commitado e pushado · py_compile verde · **sem schema/SQL/migration, sem global, sem motor paralelo** · correção mínima e conservadora.
> **Data:** 2026-06-09 · **Modelo:** Claude Opus 4.8 · **Branch:** main

## 1. Causa raiz
O upload **semântico** (Admin → Base de Conhecimento, modo "RAG Semântico") cria o registro em `public.documents` via `DocumentService.upload_document`. Esse insert listava as colunas antigas **mas NÃO definia `scope`** (coluna adicionada pelo SQL 41C.2A). Como o 41C.2A tornou `scope` (e/ou outras colunas) **NOT NULL**, o insert **falhava**. Pior: `upload_document` **engolia a exceção** (`except … return None`), então a rota recebia `None` → retornava o **erro genérico** `500 "Falha ao fazer upload do documento"`, sem o motivo real. Resultado: documento **nunca persistia** → lista permanece **Documentos (0)** e o chat não acha a palavra-chave. (O erro real ficava só nos logs da API.)

## 2. Fluxo real encontrado
- **UI** (`DocumentManagementModal.tsx`) → `POST /api/admin/proxy/documents/upload` (FormData: `file, company_id, agent_id, strategy, ingestion_mode`) → **proxy admin** (`authenticatedProxy`, `X-Admin-API-Key`) → backend `POST /api/documents/upload` (`require_master_admin`).
- Backend (`documents.py`): modo `filesystem` (.md) usa `_handle_filesystem_upload` (esse JÁ setava `scope` no 41C.2B); modo `semantic` (o usado, .txt) chama `DocumentService.upload_document` → **insert sem `scope`** → falha silenciosa → `None` → `raise HTTPException(500, "Falha ao fazer upload do documento")`.
- A UI já exibia `error.detail`, mas o detail era genérico (a causa real era abafada no service).
- **Lista:** `GET /api/admin/proxy/documents/?company_id=` (funciona; só não há docs porque o insert falhou).

## 3. Arquivos alterados
- `backend/app/services/document_service.py` — `upload_document`: adiciona **`scope`** ao insert (`normalize_document_scope(agent_id)`) e **re-lança** a exceção em vez de retornar `None` (surfacing do erro real).
- `backend/app/api/documents.py` — novo `GET /api/documents/rag-health` (probe raso: MinIO + Qdrant acessíveis, sem segredo).
- `docs/canon/design/2026-06-claude-design/41C.1.1-rag-upload-debug-report.md` (este).

## 4. Correções feitas
1. **`scope` no insert semântico** (default seguro `agent` quando há `agent_id`, senão `tenant`) → corrige a violação de NOT NULL.
2. **Não engolir o erro:** `upload_document` agora **propaga** a exceção; a rota retorna `500` com o **detalhe real** (ex.: constraint de coluna) → a UI mostra o erro verdadeiro (já lê `error.detail`). Isso também revela imediatamente se ainda faltar **outra** coluna NOT NULL (curation_status/visibility/etc.).
3. **Probe `/rag-health`** (via `/api/admin/proxy/documents/rag-health`) → `{documents_route, minio, qdrant}` para distinguir DB × MinIO × Qdrant sem ler logs.

## 5. O que NÃO foi alterado
RLS de `documents` (sem mexer), schema/SQL/migration, isolamento por `agent_id`/tenant-wide (41C.1/41C.2B intactos), `include_global` (segue **off**), conhecimento global, `autobrokers_global`, billing, sanitização, WhatsApp/Auxiliares/ResultVision, UI do modal (a UI já exibia `error.detail`; a correção foi no backend).
