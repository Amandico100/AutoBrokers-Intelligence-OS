## 1. Resumo executivo
Adicionado o **gate de saldo** ao Auxiliar de Resumo: antes de tocar no banco/LLM, o backend verifica `has_sufficient_balance(company_id)` (reuso do billing herdado). Sem saldo → **HTTP 402** `insufficient_credits`, **sem chamar LLM, sem gravar `token_usage_logs`, sem cobrar e sem criar run**. A Next API propaga 402 com erro identificável + mensagem amigável; a UI mostra um card "Sem créditos suficientes" com CTAs. **Nenhuma estrutura de cobrança nova** — débito segue assíncrono (token_usage_logs → worker).

## 3. Onde o gate foi aplicado
No endpoint `POST /api/auxiliaries/resumo-atendimentos/run` (FastAPI), **logo após** `_require_internal_key` + validação de `company_id` e **antes** de: localizar o tenant auxiliary, escolher conversa, criar `auxiliary_run`, ler mensagens e chamar a LLM. Ou seja, bloqueia **antes de qualquer custo ou efeito colateral**.

## 4. Como o saldo é verificado
Reuso **exato** do padrão de `api/documents.py` (benchmark) e `api/webhook.py` (chat):
```python
from app.services.billing_service import get_billing_service
if not get_billing_service().has_sufficient_balance(company_id):
    raise HTTPException(status_code=402, detail="insufficient_credits")
```
`has_sufficient_balance` é síncrono, com **cache de 30s** (respeitado), e consulta `company_credits` via `BillingCore`. **Não** debita nada; o débito continua assíncrono (`token_usage_logs` → worker Celery). Fail-closed: sem saldo (ou sem créditos) → bloqueia, igual ao chat.

## 5. Como o erro 402 é propagado
- **Backend:** `HTTPException(402, detail="insufficient_credits")`.
- **Next API:** após o fetch, se `response.status === 402` (ou `detail/error === 'insufficient_credits'`), retorna `{ success:false, error:'insufficient_credits', message:'Sua corretora não possui créditos suficientes para executar este auxiliar.' }` com **status 402** (não vira 500, não vaza stack trace/segredo). Demais erros mantêm o tratamento genérico anterior.
- **Tipo:** `RunResumoResponse.message?` permite a UI exibir a mensagem.

## 6. Como a UI trata saldo insuficiente
No detalhe do Auxiliar, quando `res.error === 'insufficient_credits'`, o estado `errorKind='credits'` renderiza um card amigável (ícone de alerta): **"Sem créditos suficientes para executar este Auxiliar."** + subtexto ("Recarregue os créditos da corretora ou ajuste o plano…") + CTAs **"Ver configurações"** (`/dashboard/configuracoes`) e **"Ver execuções"** (`/dashboard/auxiliares/execucoes`). Outros erros mantêm a mensagem genérica + "Tentar novamente". Sem tela de pagamento/top-up/Stripe.

## 10. Riscos/remanescentes
- **Comportamento de `has_sufficient_balance` sem subscription/credits:** empresas sem linha em `company_credits`/sem plano podem ser bloqueadas (mesmo critério do chat). Consistente e intencional.
- **Cache de 30s:** uma empresa que zera o saldo pode rodar por até ~30s antes do bloqueio (igual ao chat). Aceitável.
- **Saldo visível na UI:** não implementado (priorizamos o bloqueio). Futuro: reusar `GET /api/billing` para exibir saldo discreto no detalhe.
- **Sem cobrança de falha:** mantido (track_cost só em sucesso). O gate evita também o caso "rodar sem saldo e negativar".
