## 1. Resumo executivo
Fluxo seguro para cadastrar WhatsApp/Z-API pelo Vault: a UI envia as credenciais a uma rota **Next server-side**, que delega a um **backend seguro** (protegido por chave interna) — o backend **cifra `token`/`client_token`** (`prepare_integration_for_storage`) e grava em `public.integrations`; a `tenant_connections` é atualizada para **referenciar `integrations.id`** (`technical_ref_type='integration'`), com auditoria. Há **teste de configuração local (dry-run) que não envia mensagem**. **A rota antiga `/api/admin/integrations` não é usada.** O token nunca volta ao frontend, nunca é logado.

## 3. Rota backend criada
`POST /api/whatsapp-integrations/configure` (exige `X-AutoBrokers-Internal-Key`, padrão dos Auxiliares): valida `company_id`/`agent_id`/`provider=z-api`/`identifier`/`instance_id`/`token`; confirma que o **agente pertence à empresa**; cifra `token`/`client_token`; **upsert** em `integrations` (`on_conflict=provider,identifier`); retorna integração **sanitizada** (`identifier_masked`, `has_instance_id`, `has_client_token`, `base_url`, `is_active`) — **sem token**.
`POST /api/whatsapp-integrations/test`: carrega a integração (escopo `company_id`), roda `prepare_integration_for_runtime` (confirma que descriptografa) e valida campos. **Não chama a Z-API**; retorna `{success, dry_run:true, message}`.

## 4. Rotas Next criadas
- `POST /api/vault/connections/[connectionId]/whatsapp/configure`: auth + `company_id` da sessão; valida que a conexão é da empresa e é template `whatsapp_zapi`; resolve `agent_id` (do payload validado ou o agente principal da empresa); **delega ao backend** com a chave interna; atualiza `tenant_connections` (`technical_ref_type='integration'`, `technical_ref_id`, `status='connected'`, `connection_config` sem segredo, `metadata.safe_secret_flow`); grava auditoria. **O token só transita server→backend; nunca é salvo no Next nem retornado.**
- `POST /api/vault/connections/[connectionId]/whatsapp/test`: valida posse + que a conexão referencia uma integração; chama o backend `/test`; grava auditoria.

## 5. Como o token é criptografado
Reuso do **39A4.1**: o backend chama `prepare_integration_for_storage(data)` → `encrypt_integration_secret` (Fernet/AES via `EncryptionService`) em `token` e `client_token` antes do upsert. Em leitura/teste, `prepare_integration_for_runtime` descriptografa **em memória**. Nenhuma criptografia foi duplicada em TypeScript.

## 6. Como tenant_connections referencia integrations
Após o backend gravar a integração cifrada, a rota Next atualiza a `tenant_connections`:
`technical_ref_type='integration'`, `technical_ref_id=<integration.id>`, `status='connected'`, `connection_config={provider:'z-api', identifier_last4, has_client_token}`, `metadata={configured_via:'vault', safe_secret_flow:true}`. **Sem segredo** em `connection_config`/`metadata`.

## 7. Como a UI configura WhatsApp
Em **Conectores → Minhas conexões**, conexões do template `whatsapp_zapi` ganham **"Configurar com segurança"** (ou "Reconfigurar"). O `ConfigureWhatsAppModal` pede telefone/Instance ID/Token/Client Token (opcional)/Base URL (opcional), com avisos ("credenciais criptografadas no servidor"; "nenhuma mensagem será enviada"). Os campos de Token/Client Token são `type="password"`, `autoComplete="off"`, **limpos do estado após salvar/fechar**; o token salvo **nunca** é renderizado/pré-preenchido. Sem `console.log` de payload.

## 8. Como o teste local funciona
Botão **"Testar"** (aparece quando a conexão está configurada) → `POST .../whatsapp/test` → backend valida que a integração existe e **descriptografa** (config válida) + campos presentes. **Nunca** chama a Z-API. Retorna mensagem amigável ("Configuração local válida. Nenhuma mensagem foi enviada.").

## 9. Como auditoria é gravada
`vault_audit_log`: `whatsapp_configured` (`action=configure_zapi`, `status=connected`, `risk=high`) no configure; `whatsapp_tested` (`action=test_config`, `status=success|failed`, `risk=medium`, `metadata.dry_run`) no teste. Sem segredo no `metadata`.
