# 39A4.3 — WhatsApp HITL Dry-Run Send Report

## 1. Resumo executivo
Implementada a mecânica central do produto: **aprovação humana → execução controlada → provider WhatsApp → auditoria**, em **dry-run forçado** (nenhuma mensagem real). Uma aprovação WhatsApp **aprovada** pode ser "executada": a rota Next valida tudo e chama o backend, que roda o `ZApiProvider` com `force_dry_run=True` (determinístico, **não depende** do env global e **não chama a Z-API**). A aprovação vira `executed` (ou `failed`) e os eventos `whatsapp_dry_run_executed`/`whatsapp_dry_run_failed` são auditados.

## 3. Rotas backend criadas/alteradas
`POST /api/whatsapp-integrations/send-dry-run` (chave interna): valida payload (`company_id`, `integration_id`, `to_number`, `message`), carrega a integração (escopo empresa+id+ativa), descriptografa em memória (`prepare_integration_for_runtime`), confirma `instance_id`/`token`/`identifier`, e chama `get_zapi_provider().send_text(..., force_dry_run=True)`. Retorna `{success, provider, dry_run:true, message}`. **Sem token na resposta/log; telefone mascarado.** O provider ganhou `force_dry_run` para simular **sem** `requests.post`, independente do `DRY_RUN` global.

## 4. Rotas Next criadas
`POST /api/vault/approvals/[approvalId]/execute`: resolve sessão/empresa; carrega a `approval_request` (escopo empresa); exige `status='approved'` e `action_type ∈ {whatsapp_draft_message, whatsapp_send_message_dry_run}`; exige `tenant_connection_id`; carrega a `tenant_connection` (empresa + `technical_ref_type='integration'` + `technical_ref_id`); extrai `to_number`/`message` de `request_payload`/`preview`; chama o backend `send-dry-run`; atualiza a aprovação para `executed` (ou `failed`) e audita.

## 5. Como a aprovação é validada
Escopo por `company_id` da sessão (anti-IDOR); `status='approved'` (409 caso contrário); `action_type` na allowlist; `tenant_connection_id` presente; conexão pertence à empresa e referencia uma integração; `to_number`+`message` presentes (erro amigável se faltarem). O update de sucesso usa `.eq('status','approved')` (idempotência: não reexecuta um já executado).

## 6. Como a conexão/integration é resolvida
A aprovação aponta para `tenant_connection_id`; a conexão referencia `integrations.id` via `technical_ref_id`. O backend carrega a integração por `company_id + id` e descriptografa **em memória**. **O Next nunca vê token.**

## 7. Como o dry-run é garantido
**Determinístico, em 2 camadas:** (a) a rota Next chama `send-dry-run` (nunca a rota de envio real); (b) o backend chama `provider.send_text(..., force_dry_run=True)` → o `_post` retorna `SendResult(dry_run=True)` **antes** de qualquer `requests.post`. Não depende do env `DRY_RUN`. Scan confirma: **sem `requests.post`/`send-text` na rota** `whatsapp_integrations.py` e em `app/api/vault`.

## 8. Como a UI executa simulação
`/dashboard/personalizacao/conectores/aprovacoes`:
- **pending** → Aprovar/Rejeitar.
- **approved** → aviso "modo seguro: nenhuma mensagem real será enviada" + **"Executar simulação"** → `POST .../execute`.
- **executed** → "Simulação executada — nenhuma mensagem foi enviada".
- **failed** → erro amigável (`error_message`).
"Criar aprovação de teste" agora resolve a **conexão WhatsApp conectada** (via templates+connections), pede o **telefone de destino** (campo inline, sem hardcode) e cria `whatsapp_send_message_dry_run` com `request_payload {to_number, message, dry_run:true}`.

## 9. Como auditoria é gravada
`vault_audit_log`: `whatsapp_dry_run_executed` (`status=success`) ou `whatsapp_dry_run_failed` (`status=failed`), `action='send_message_dry_run'`, `risk` herdado. `metadata`: `provider`, `dry_run`, `tenant_connection_id` (via coluna), `approval_request_id`, `message_length`, `to_last4`. **Sem token/instance_id/client_token.**

## 10. O que NÃO foi implementado
Envio real, QR, Evolution, n8n, Auxiliar de Cobrança, portais. Sem SQL/migration/schema. Sem telefone hardcoded. Não dependemos do `DRY_RUN` global para segurança.

## 12. Como testar manualmente (pós-deploy API + Web)
1. Garanta a conexão WhatsApp **connected** no Vault (39A4.2).
2. `/dashboard/personalizacao/conectores/aprovacoes`.
3. Em "Criar aprovação de teste (WhatsApp dry-run)", informe um **telefone de destino** → "Criar aprovação de teste".
4. **Aprovar** o pedido.
5. **Executar simulação** → status **executed** + "nenhuma mensagem foi enviada".
6. **Auditoria** (`/conectores/auditoria`) mostra `whatsapp_dry_run_executed`.
7. Confirme que **nenhuma** mensagem chegou ao WhatsApp.
