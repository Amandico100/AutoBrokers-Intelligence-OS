## 1. Resumo executivo
O **Smith já tem uma base sólida de RAG por corretora** (documents → MinIO → chunking → embeddings → Qdrant `company_{id}` → retrieval via LangChain) e de **memória** (session_summaries, user_memories, memory_settings, conversation_logs). **Faltam 3 coisas para o que queremos:** (1) **conhecimento GLOBAL** (não existe — tudo é per-company); (2) **isolamento por agente no retrieval** (hoje filtra só `company_id` → P0 de vazamento entre agentes da mesma corretora) + **RLS policies em `documents`** (RLS ligado, zero policies); (3) **separação clara entre conhecimento estruturado vs RAG textual** (boa parte do legado é regra/contrato/skill = **estruturado**, não documento RAG). O legado (AGENT_OS_WORKSPACE + ResultVision) é **ouro de domínio** (corredores, fases, skills, guardrails, templates, recipes de seguradora, golden tests) e deve ser **migrado com curadoria** como SPEC/estruturado/RAG-curado — **nunca como runtime** e **sem PII/segredos**. **Não criar motor RAG paralelo:** usar o do Smith e estendê-lo.

## 2. Inventário do RAG atual no Smith
- Tabela **`documents`**: `id, company_id (NOT NULL), agent_id (nullable), file_name/type/size, minio_path, qdrant_collection (nullable), status(pending/processing/completed/failed), chunks_count, ingestion_strategy(recursive|semantic|page|agentic|csv), ingestion_mode(semantic|filesystem), quality_score/quality_audited_at, error_message, fs_storage_path/fs_token_count/fs_section_count/fs_outline(jsonb), created_at/updated_at/processed_at`. **RLS ENABLED, zero policies** → isolamento é só na aplicação (P0).
- Serviços: `document_service.py`, `ingestion_service.py`, `qdrant_service.py`, `minio_service.py`, `langchain_service.py`, `benchmark_service.py`, `sanitization` (jobs).
- Sem conceito de **escopo/global**; `agent_id` opcional mas usado como filtro de payload no Qdrant.

## 3. Fluxo atual de upload
`POST /documents/upload` (`require_master_admin`) → **gate de saldo** (`has_sufficient_balance`) → `agent_id` **obrigatório** → 2 caminhos: **semantic** (PDF/DOCX/TXT → MinIO → background → ingestion → Qdrant) ou **filesystem** (`.md`, 1:1 por agente, sem Qdrant, busca por outline). Rate-limit 30/min. Admin vincula KB→agente via `DocumentManagementModal`.

## 4. Fluxo atual de chunking/embedding
`ingestion_service.py`: lê raw JSON do MinIO → estratégia de chunk (recursive 1000/200; semantic por percentil; page; agentic via gpt-4o-mini com cost callback; csv linha→chunk). Safety valve 1900 chars. **Dense:** `text-embedding-3-small` (1536). **Sparse:** BM25 (fastembed `Qdrant/bm25`). Custo rastreado (`usage_service.track_cost_sync`). Insere no Qdrant com `agent_id` no payload.

## 5. Uso atual de Qdrant
`qdrant_service.py`: coleção **por empresa** `company_{company_id}` (sem coleção global). Índices: `document_id`, `agent_id`, `metadata.file_type`. Busca **híbrida** dense(cosine)+sparse(BM25) com RRF, `score_threshold≈0.4`. `agent_id` disponível para filtro no payload.

## 6. Uso atual de MinIO
Bucket `documents`; path `{company_id}/{document_id}/{filename}` + `{company_id}/raw/{document_id}.json`. **Sem ACL por objeto** (isolamento por path + rede). Filesystem mode usa `fs_storage_path`.

## 7. Uso atual de LangChain / document retrieval
`langchain_service.search_documents(query, company_id, top_k=3)` → embed query → `qdrant.search_similar(company_id, ...)`. **No chat, o filtro é só `company_id`** (`langchain_service.py:515-554`) → **`agent_id` NÃO é aplicado no retrieval** (vazamento potencial entre agentes da mesma empresa). Guardrails de segurança rodam antes; histórico vem do Supabase; grafo LangGraph cacheado por `updated_at`.

## 8. Como Agents usam conhecimento hoje
`agents`: `retrieval_mode (semantic|filesystem)` (default semantic), `is_hyde_enabled` (default false), `allow_vision`/`vision_model`, `is_subagent`, `allow_direct_chat`. RAG é ligado por `retrieval_mode`. HyDE usado em benchmark. Documentos vinculados por `documents.agent_id`.

## 9. Como SubAgents podem usar conhecimento
SubAgents são `agents` (mesma tabela/colunas) → **podem ter `retrieval_mode` + documentos próprios** (41A.3 já expôs as abas Memória/Segurança para subagents). **Pré-requisito:** corrigir o filtro `agent_id` no retrieval para o especialista ver **apenas** seus documentos.

## 10. Como Auxiliares Globais se conectam a conhecimento
Hoje: o **blueprint** (`auxiliary_templates.default_config.runtime.agent_blueprint`) pode declarar `retrieval_mode`; ao instalar, cria um **agent local por corretora** (41A.1/41A.2) que recebe **documentos locais** dela. **Falta:** declarar **conhecimento GLOBAL requerido** pelo Auxiliar (refs curadas) e injetá-lo na instalação/retrieval. Proposta na §23.

## 21. Global vs Carrier vs Tenant vs Agent vs Workflow vs Case Memory
- **Global** (AutoBrokers + Carrier): **read-only curado**, versionado, compartilhado entre corretoras; **nunca contém dado de corretora**.
- **Tenant/Agent:** isolado por `company_id` (+ `agent_id`); RAG existente.
- **Workflow/Corredor:** estruturado, referenciado por agentes/orquestrador.
- **Case/Conversation:** memória operacional **separada** do conhecimento (não vira RAG geral).

## 22. Riscos de vazamento/mistura de conhecimento
1. **`documents` sem RLS policy** (RLS on, 0 policy) — **P0**. 2. **Retrieval sem filtro `agent_id`** (vazamento entre agentes da mesma corretora) — **P0**. 3. **MinIO sem ACL por objeto** (depende de rede). 4. **Misturar case memory/PII no RAG** (LGPD) — proibir. 5. **Global contaminado com dado de corretora** — proibir (curadoria). 6. **Qdrant Cloud API key** em env plano (rotação).

## 23. Recomendação de arquitetura
- **Não criar motor paralelo.** Estender o RAG do Smith:
  1. **Global knowledge:** adicionar `scope` (`global|tenant|agent`) em `documents` (+ `valid_until`, `version`, `source`, `category`) e uma **coleção Qdrant `global`** read-only curada. (Ponte temporária possível: uma "company AutoBrokers Global", mas recomendamos `scope` estruturado.) → **SQL controlado do Architect (SPEC-003/41C.1), não neste batch.**
  2. **Segurança P0:** criar **RLS policies** em `documents` (company-scoped) + **aplicar filtro `agent_id`** no `search_documents` quando o agente for especialista; **ordem de retrieval**: agent → tenant → global curado (case memory à parte, nunca como fonte factual de cobertura).
  3. **Conhecimento estruturado** (guardrails/contratos/skills/fases/recipes) vive como **SPEC/JSON/blueprint/connector_templates** — não como documento RAG.
  4. **Auxiliar/Agent:** blueprint declara `requires_knowledge` (refs globais por slug/versão); instalação cria agent local + injeta refs globais + recebe docs locais; SubAgent especialista usa `retrieval_mode` + seus docs.
  5. **Corredores:** AGENT_OS corridor specs → `runtime.kind=workflow` (futuro), orquestrados pelo orquestrador Smith + especialistas; ação externa via **Vault + HITL** (já existe).

## 24. O que deve ser estruturado vs RAG textual
- **Estruturado (não RAG):** guardrails, contratos/schemas, skills, fases de corredor, recipes/sinais de seguradora, templates (registry), regras de canal, ownership/source-of-truth.
- **RAG textual:** condições gerais/coberturas de seguradora (prosa longa), manuais/playbooks em prosa, documentos da corretora, documentos do agente.
- **Nunca RAG:** PII, mensagens completas de clientes, segredos, logs crus.

## 25. Modelo de versionamento
- Conhecimento **global** com `version` + `valid_until` + `source` + `status` (draft/active/retired) — reaproveitar a **status taxonomy** e o **load_policy** do AGENT_OS. Atualizar conhecimento global **não** sobrescreve personalização local (docs do tenant/agent são independentes). Blueprint referencia **versão** de conhecimento global.

## 26. Modelo de permissões
- **Global:** leitura por todos os agentes (curado); escrita só Admin master. **Tenant:** isolado por `company_id` (RLS policy). **Agent:** filtro `agent_id` no retrieval. **Connector/segredos:** só Vault (cifrado). **Auditoria:** `vault_audit_log` para ações externas; `conversation_logs` para retrieval traces.
