# 41C.1 — RAG Security P0: Documents RLS + Agent-Scoped Retrieval Report

> **Status:** concluído, commitado e pushado · py_compile verde · **sem schema novo, sem coluna nova, sem global, sem migration auto-aplicada** · SQL de RLS é **manual**.
> **Data:** 2026-06-09 · **Modelo:** Claude Opus 4.8 · **Branch:** main

## 1. Resumo
Fechados os 2 gaps P0 do RAG do 41C.0: (A) **policies RLS** em `public.documents` (defesa em profundidade — script **manual**, idempotente, espelha `integrations`/`leads`); (B) **retrieval por agente** — descobrimos que o **caminho de produção já filtra `agent_id`** (knowledge_base tool → `SearchService` → `qdrant.search_similar(agent_id=…)`); o gap real era o método **legado** `search_documents`/`get_rag_context` (sem callers) + a **ausência de docs tenant-wide** (agent_id NULL). Hardening: legado agora é agent-scoped; e o caminho vivo passa a incluir **docs tenant-wide** (agente vê os seus + compartilhados, **nunca de outro agente**). Sem global, sem coluna nova.

## 2. SQL criado e o que ele faz
`docs/sql/41C1-documents-rls-policies.sql` (**manual, NÃO executado**): `enable row level security` + 2 policies idempotentes (`drop policy if exists` antes): **Service role full access** (backend usa service_role) e **"Users can manage own company documents"** para `authenticated` (`company_id = (select u.company_id from users_v2 u where u.id = auth.uid())`, USING + WITH CHECK). Termina com SELECTs de verificação (rls_enabled, policies, count). Como `documents` count = 0, **zero risco** a dados existentes.

## 3. Como a RLS ficou desenhada
- **service_role:** acesso total (o backend opera por service_role; RLS é defesa em profundidade).
- **authenticated:** só a própria empresa (anti cross-tenant), resolvendo `company_id` via `users_v2.company_id` do `auth.uid()` — exatamente o padrão de `public.integrations`/`public.leads` já presente no banco.
- **Sem** policy global/scope (não é deste batch).

## 4. Como o retrieval por agent_id ficou desenhado
- `qdrant_service.search_similar` ganhou `include_tenant_wide: bool = False`. Com `agent_id`:
  - `include_tenant_wide=True` → filtro **OR**: `agent_id == X` **OU** `agent_id` ausente (tenant-wide) — via `should=[FieldCondition, IsEmptyCondition]`. **Nunca** retorna docs de outro agente.
  - `include_tenant_wide=False` (default) → comportamento estrito atual (inalterado).
- **Degradação segura:** `IsEmptyCondition`/`PayloadField` importados com `try/except` (`_HAS_IS_EMPTY`). Se o client Qdrant não tiver, cai para o filtro estrito (sem quebrar o RAG).
- `langchain_service.search_documents`/`get_rag_context` (legado) agora aceitam `agent_id` + `include_tenant_wide=True` e repassam ao Qdrant.
- `SearchService._execute_search` (caminho **vivo** do chat) passa `include_tenant_wide=True` → agente principal e subagent veem **seus docs + tenant-wide**, isolados de outros agentes.

## 5. Compatibilidade com tenant-wide docs (agent_id IS NULL)
- Hoje `documents.agent_id` é obrigatório no upload, então não há docs tenant-wide ainda; a capacidade fica **pronta** para quando criarmos docs compartilhados por corretora.
- Chamadas **sem** `agent_id` continuam company-wide (inalterado).
- `benchmark_service` (busca por documento específico/coleção) **não** foi alterado.
